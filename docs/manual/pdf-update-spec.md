## C. v0.5.1 내용 (rm 동작 정형화) — 04·05·06장

- **04장** `/init` 근처 콜아웃 `v0.5.1 NOTE` 추가: 토큰 작업(`/init`·`/ds-token`)은 끝나면 sentinel을 에이전트가 **직접 node로 정리**(사용자 `rm` 불필요, `rm`은 deny). 완료 "다음 단계"는 **고정 메뉴**(Figma 키 있으면 `/figma-tokens`, 없으면 `/ds-component` + `pnpm eval:baseline`·`pnpm dev`).
- **05장** "토큰 파일 수정 차단" 항목 끝에 한 문장: v0.5.1부터 `/init`·`/ds-token`이 sentinel을 직접 node로 정리하므로 사용자가 칠 일 없음. 에이전트가 `rm` 제안하면 구버전 동작.
- **06장** 헌법 설명에 규칙 2줄: ① 임시파일·sentinel 정리는 항상 node(`rm` 금지) ② 완료 "다음 단계"는 커맨드 .md 고정 메뉴만.

## D. v0.6.0 신규 — 화면설계서 `/screen-spec`

### D-1. 04장 명령어 표에 행 1개 추가
| 명령 | 트랙 | 역할 | 전제 |
|---|---|---|---|
| `/screen-spec <화면>` | 화면 | 화면설계서(인터랙션 명세) — 흐름 + 콜아웃 + Screen ID + Description 표 | write MCP |

### D-2. 04장에 소절 신규 — "화면설계서 — /screen-spec (v0.6 신규)"
와이어(`/wireframe`, 회색 정보구조) **다음 단계**. 무엇을 누르면 어떻게 동작하는가를 Figma에 명세서로. 담당 `screen-spec-builder` + `screen-spec-template` 스킬.

```
/screen-spec 상품옵션선택
# 1 screen-spec-template 스킬 + DESIGN.md/PROJECT.md 로드
# 2 이번 화면만 1회 확인 (플로우 스텝·Application to) — 양식은 안 물음
# 3 Screen ID 생성 (추론, 애매하면 1회)
# 4 메타헤더 + 폰 목업 흐름(DS 컴포넌트 재사용) + 오렌지 콜아웃 + Description 표
# 5 get_screenshot 검증
```

- **Screen ID**: `{PRJ}-{PLAT}-{CH}-{TYPE}-{SEQ}` (예: `PRD-MO-1-PG-001`)
- **색 규칙**: 오렌지=콜아웃·화살표·헤더점선 / 초록=`※`참조주 / 블루=인터랙션점·포커스 / 핑크=APP / 탄=MOBILE WEB / 회색=목업. (목업은 회색만, 주석만 색)
- **명시 vs 되묻기**: 레이아웃·색·표·ID 포맷은 스킬에 고정, **이번 화면 플로우만** 1회 확인.

### D-3. (선택) KEY INSIGHT 박스
`/wireframe`은 "무엇이 어디에"(정보구조), `/screen-spec`은 "무엇을 누르면 어떻게"(동작). 보통 `/wireframe` → `/screen-spec` → `/ds-component` 순.

### D-4. 06장 "커스터마이징할 만한 곳" 보강(선택)
새 트랙을 붙인 실례로 `/screen-spec`(새 커맨드+에이전트+스킬)을 언급 — "에이전트가 또 생기는 정상적 확장".

## E. PROJECT.md 관련 (07장/부록에서 언급 시)
PROJECT.md에 `## 화면설계서` 섹션 추가됨 (Screen ID 포맷·플랫폼 코드·Spec 페이지 ID). 매뉴얼에서 PROJECT.md 항목을 나열하는 곳이 있으면 한 줄 추가.
