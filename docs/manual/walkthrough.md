# 실전 워크스루 — `/init`부터 첫 컴포넌트까지 (왜 · 어떻게)

> 이 장은 **처음 쓰는 사람이 이 글만 보고 따라올 수 있게** 만든 실전 기록이다.
> 각 단계를 **① 무엇을 한다 → ② 안에서 어떻게 도는가 → ③ 실제로 뭐가 나오는가 → ④ 왜 하는가** 순으로 설명한다.
> 숫자·출력은 실제 프로젝트(브랜드색 초록 `#22c55e`)에서 그대로 가져왔다.
>
> 매뉴얼 위치 제안: 02장(5분 시작)과 03장(결과물) 사이의 **"따라하기" 장**.

---

## 0. 시작 전에 — 머릿속에 3개만 넣자

### (1) 누가 누구를 부르나
```
당신 → 커맨드(또는 자연어) → 에이전트 → 스킬·도구
                                ↑ 훅이 옆에서 감시(잘못하면 차단)
```
- **커맨드**(`/init` 등)는 *입구*다. 치면 **에이전트**(실제 일꾼)가 깨어나 일한다.
- 에이전트는 끝나면 **"다음은 이거 하세요"**라고 추천만 한다. **단계는 한 번에 하나** — 당신이 다음 신호를 준다. (자동으로 쭉 안 굴러간다 = 중간에 검토·결정할 수 있다)

### (2) "토큰"이 뭐고 왜 중요한가
**토큰 = 디자인 값에 붙인 이름.** `#22c55e` 같은 raw 값 대신 `--color-primary`라는 *이름*을 쓰는 것.
- 색을 바꿔야 하면 → **토큰 한 줄**만 고치면 그 토큰 쓰는 **모든 컴포넌트가 자동으로** 바뀐다.
- 코드·Figma가 **같은 이름**을 쓰면 디자이너와 개발자가 같은 언어로 말하게 된다.
- 이름에 **접근성·다크모드 규칙**을 박아두면, 매번 신경 안 써도 자동으로 지켜진다.
> 핵심: **하드코딩(`#22c55e`)을 흩뿌리지 않고, 이름(토큰) 하나로 통제**하는 것. 그래서 토큰을 먼저 만든다.

### (3) 두 갈래 트랙
- **코드 트랙**: `/ds-component` 등 → 코드(`src/`)를 만든다.
- **Figma 트랙**: `/build-component`·`/figma-tokens` → Figma에 실제로 만든다. (write 가능한 Figma MCP 있을 때)
- 둘은 `/ds-token`에서 만나 서로 맞춘다.

## 1. STEP 0 — `PROJECT.md` (씨앗 심기)

### ① 무엇을
프로젝트당 **딱 한 번** 채우는 파일. 브랜드색 1개, 폰트, (선택) Figma 키/페이지 ID, 톤.

### ④ 왜
이 한 장이 **모든 자동 생성의 씨앗**이다. 브랜드색 **하나**만 넣으면 거기서 전체 색 스케일이 나온다. "사람의 결정"은 여기(=무엇을 만들지)서 끝나고, 이후 "어떻게"는 전부 자동이다.

```
- 브랜드 컬러 (hex 1개): #22c55e
- 폰트: Pretendard
- Figma 파일 키 / 페이지 ID: (Figma 트랙 쓸 때만)
- 톤: "노션처럼 정돈된 정보 밀도" 같은 구체적 명사구 3~5개
```
> 비어 있으면 에이전트가 "일반 기본값"으로 일해서 "AI가 만든 듯한" 결과가 나온다. **톤·금지 규칙**을 채우는 게 품질을 가장 크게 바꾼다.

## 2. STEP 1 — `/init` (토큰 생성 + 카탈로그 초기화)

### ① 무엇을
`PROJECT.md`를 읽어 **코드 디자인 토큰(`src/tokens/*.css`)을 생성**하고 `DESIGN.md` 카탈로그를 만든다. **프로젝트당 한 번.**

### ② 어떻게 도는가 (안에서)
`/init` 커맨드는 곧장 **`ds-token-builder` 에이전트**에게 위임한다. 에이전트는:
```
1  PROJECT.md 파싱 (브랜드색·폰트·다크모드 여부)
2  color-scale-builder 스킬로 브랜드색 → OKLCH 50~900 스케일 생성
3  neutral 스케일(무채색) + 시맨틱색(success/warning/danger/info) 생성
4  src/tokens/*.css 7개 파일 작성
5  src/styles.css 에서 토큰 import 주석 해제
6  WCAG 대비 자동 검증 (미달이면 주석/조정)
7  DESIGN.md 카탈로그 작성
8  sentinel 정리(node) + 보고  ← v0.5.1: 에이전트가 직접, rm 안 씀
```

### ③ 실제로 나온 것 (이번 프로젝트)
```
/init 완료. 토큰 7개 파일 생성, WCAG AA 통과, 타입체크 클린.
- brand: #22c55e → brand-600 베이스 (초록 149.6°, OKLCH 균일)
- success-600 / warning-400(검정 텍스트로 AA 확보) / danger-600 / info-600
- light/dark 양쪽 시맨틱 alias 정의
```
생성된 파일과 역할:
| 파일 | 담는 것 |
|---|---|
| `colors.css` | brand 50~900, neutral 0~900, semantic alias(`--color-primary` 등), light/dark |
| `typography.css` | `--font-sans`, `--font-size-*` |
| `spacing.css` | `--space-*` (4px 그리드) |
| `radius.css` | `--radius-*` |
| `motion.css` | duration·easing |
| `semantic.css` | focus-ring·shadow·z-index |
| `index.css` | 위 전부 import |

### ④ 왜 이렇게 하나 (핵심)
- **brand 1개 → 전체 스케일**: 50~900을 손으로 안 고른다. OKLCH로 **밝기 단계가 균일**하게 계산돼, 어디에 써도 톤이 고르다.
- **raw vs semantic alias 2층 구조**:
  - raw = `--color-brand-500` (실제 색)
  - alias = `--color-primary` → `brand-600`을 가리킴
  - 컴포넌트는 **alias만** 쓴다. 그래서 "primary를 한 단계 어둡게" 같은 변경이 **alias 한 줄**로 끝난다.
- **WCAG 자동 검증**: 초록 brand 위 흰 텍스트가 대비 부족하면, 그래서 primary를 `brand-600`으로(한 단계 어둡게) 자동 선택했다. warning(노랑)은 검정 텍스트라야 대비가 나오니 그렇게 잡았다. → **접근성을 사람이 매번 계산 안 해도 됨.**
- **DESIGN.md**는 에이전트가 자동 갱신하는 카탈로그다. **직접 편집 금지**(다음 실행 때 덮어써짐). 정의를 바꾸려면 `PROJECT.md`를 고친다.

> 확인 팁: `pnpm eval` 을 돌리면 토큰 대비가 **CRITICAL 0건**인지 점수로 나온다.

## 3. STEP 2 — `/figma-tokens` (토큰을 Figma에 실제 생성)

### ① 무엇을
방금 만든 코드 토큰을 **Figma Variables로 실제로 만든다.** (write Figma MCP 필요) 담당: `ds-figma-token-builder`.

### ④ 왜
- 디자이너는 Figma에서 작업한다. 코드의 토큰과 **이름이 1:1로 같은** 변수를 Figma에도 두면(`--color-brand-500` ↔ `brand/500`), **둘이 같은 언어**가 된다.
- 이후 컴포넌트를 Figma에서 그릴 때 raw 색을 찍지 않고 **이 변수를 바인딩**한다 → 코드와 안 어긋난다.

### ③ 실제로 나온 것
```
컬렉션 4개 / 신규 변수 83개
- Colors / Typography / Spacing / Radius
- swatch 시각 검증 PASS (Light 356:2 / Dark 360:2)
주목: Figma에 기존 brand가 주황(#f97316)으로 잘못 들어가 있던 걸
      PROJECT.md 기준 초록(#22c55e)으로 재기록.
```
> 이 "주황→초록" 보정이 바로 다음 단계가 필요한 이유다 — 코드와 Figma가 어긋나 있었다(드리프트).

## 4. STEP 3 — `/ds-token` (코드 ↔ Figma 정합 맞추기)

### ① 무엇을
코드 토큰과 Figma 변수를 **양방향 비교**해서 어긋난 곳을 분류하고, 사람이 결정해 맞춘다.

### ② 분류
| 분류 | 뜻 | 조치 |
|---|---|---|
| 일치 | 양쪽 같음 | 무시 |
| Figma에만 | 코드에 없음 | 추가할지 묻는다 |
| 코드에만 | Figma에 없음 | 추가할지/폐기인지 |
| **값 충돌** | 양쪽 있는데 값 다름 | **사람이 결정** |

### ③ 실제로 나온 것
- 처음: **충돌 33건** (신호색 success/warning/danger + radius)
- 결정 기준:
  - **색·대비 토큰 → 검증된 코드(OKLCH) 채택** (코드는 `/init`에서 WCAG 통과한 값)
  - **radius 같은 미적 토큰 → 의도된/풍부한 쪽 채택** (xs=2px 추가, 2xl 20→24px)
- 재검증: **충돌 0건** (색 70/70 일치, radius 8/8, font-size 6/6, 코드전용 12개는 의도)

### ④ 왜 충돌이 "정상"인가
`/ds-token`의 **존재 이유가 "어긋난 곳을 찾는 것"**이다. 충돌이 떴다 = **드리프트를 제대로 감지**했다는 뜻. 위험한 건 충돌이 안 뜨고 **조용히 어긋난 채 쓰는** 것이다. 처음엔 코드와 Figma가 따로 만들어져서 많이 갈리지만, 한 번 맞추고 `/figma-tokens` 재실행하면 이후엔 대부분 "일치"로 조용해진다. (= **최초 1회 정합 작업**)

## 5. STEP 4 — 컴포넌트 (`/build-component` + `/ds-component`)

### ① 무엇을
- `/build-component Button` → **Figma에** Component Set + 문서 생성
- `/ds-component Button` → **코드에** cva 래퍼 + 스토리 생성

### ③ 실제로 나온 것 (Button)
```
Figma: Button Component Set (intent 4 × size 3 × state 3 = 36 variants)
       + Dark Preview, Light/Dark 시각 검증 PASS
       모든 컬러·라디우스·타이포·스페이싱이 Figma Variables에 바인딩
코드 : src/components/Button/ (cva 래퍼 + 스토리), DESIGN.md 카탈로그 갱신
       (참고: Pretendard가 Figma에 없어 Inter로 폴백 — 폰트 업로드 시 교체)
```

### ④ 왜 둘 다
- Figma 본 = 디자이너가 보는 진실, 코드 래퍼 = 개발이 쓰는 진실. **둘의 variant 이름을 일치**시켜 어긋남을 막는다.
- 보통 `/build-component`(디자인) → 확인 → `/ds-component`(코드) 순. 코드 먼저 만들면 그 cva 구조를 Figma가 미러링하게 할 수도 있다.
- 둘 다 **STEP 1~3에서 만든 토큰**을 alias로 쓴다 → 그래서 토큰을 먼저 만든 것이다.

## 6. STEP 5 — `/qa-a11y` (접근성 검증)

### ① 무엇을
컴포넌트의 **모든 색 조합 + 다크모드 동등성 + ARIA/키보드**를 WCAG 2.2 AA로 검사.

### ③ 실제로 나온 것 (Button)
```
A11y 32/32 PASS — CRITICAL 0, 다크 동등성 위반 0
primary 5.32 / 6.49 (L/D), danger 6.66 / 5.55, secondary text 18.1 / 17.34 …
```

### ④ 왜
브랜드색이 초록으로 바뀌면 대비가 달라진다. **눈으로 멀쩡해 보여도 대비가 미달**일 수 있다(토스 블루 #3182f6 위 흰 텍스트가 3.71:1로 AA 미달인 것처럼). QA가 이런 함정을 전부 잡고, 미달이면 **OKLCH 조정안**까지 제시한다.

## 7. 전체 플로우 한눈에 (그리고 "왜 이 순서")

```
PROJECT.md  ─(씨앗)→  /init  ─(코드 토큰)→  /figma-tokens  ─(Figma 토큰)→  /ds-token
   │                                                                          │(정합)
   └──────────────────────────────────────────────────────────────────────────┘
                                      ↓
                  /build-component(Figma) + /ds-component(코드)  ──→  /qa-a11y / /qa
                                      ↓
                              self-check → pnpm eval (하네스 회귀 점검)
```
**왜 토큰 → 컴포넌트 → 화면 순서?** 컴포넌트는 토큰(색·간격·라디우스)을 *재료*로 쓴다. 재료가 없으면 컴포넌트를 못 만든다. 그래서 **토큰이 항상 먼저**다.

## 8. 자주 나오는 "왜?" 모음

**Q. 왜 `#22c55e`를 직접 안 쓰고 토큰을 쓰나?**
색 한 번 바꾸려면 코드 전체를 뒤져야 하기 때문. 토큰이면 한 줄로 끝나고, 훅이 하드코딩을 아예 차단한다.

**Q. 왜 raw 말고 alias(`--color-primary`)를 컴포넌트에서 쓰나?**
"primary를 어둡게" 같은 의미 단위 변경을 alias 한 줄로 하기 위해. raw를 직접 쓰면 그 색을 쓰는 곳을 다 고쳐야 한다.

**Q. 왜 충돌이 그렇게 많이 떴나? 잘못한 건가?**
아니다. 코드와 Figma가 따로 만들어졌다 처음 맞대면해서 그렇다. 도구가 드리프트를 **제대로 잡은** 것 = 건강한 신호. 최초 1회만 많고 이후엔 조용하다.

**Q. 왜 한 번에 다 자동으로 안 돌아가나?**
중간에 **사람이 결정할 게이트**(색 충돌, 브랜드 드리프트, QA CRITICAL)가 있어서. 자동조종이면 그걸 멋대로 잘못 결정한다. 그래서 **한 단계씩 + 게이트에서 멈춤**이 의도된 설계다.

**Q. Figma 없이도 되나?**
된다. `/init`·`/ds-component`·`/qa`·`/wireframe`·`pnpm eval`은 Figma 없이 전부 작동한다. Figma 트랙(`/figma-tokens`·`/build-component`)만 write MCP가 있을 때 돈다.
