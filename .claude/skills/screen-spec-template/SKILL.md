# Screen Spec Template — 화면설계서 양식 정의

> 이 스킬은 **양식·규칙(매번 같음)을 명시**한다. **이번 화면의 값(매번 다름)만** 에이전트가 되묻는다.
> 목표: 누가 만들어도 동일한 양식의 화면설계서가 나오게.

## 0. 무엇인가 — 와이어프레임과의 차이
- `wireframe`(wireframe-grayscale): **회색 정보구조** 화면 (무엇이 어디에)
- **screen-spec(이 스킬)**: 그 다음 단계 — **인터랙션 명세** (무엇을 누르면 어떻게 동작하는가) + 화면 흐름 + Screen ID + Description 표
- 산출 타깃: **Figma 프레임** (write MCP 필요). 없으면 코드/HTML 스펙시트로 폴백.

## 1. 레이아웃 영역 (고정)

```
┌───────────────────────────────────────────────────────────────────────────┐
│ [제목바]  <화면명>                                              </>(dev icon)│
├──────────────┬──────────────────────────────┬───────────────┬──────────────┤
│ [메타-좌]     Chapter / Screen Path           [메타-우]        [Description] │
│              (케이스 경로)                     Application to   Num│Desc 표   │
│                                               Screen ID                      │
├───────────────────────────────────────────────┴───────────────┴────────────┤
│ [캔버스] 폰 목업들의 흐름 (→ 점선 화살표) + 콜아웃 번호 + 인터랙션 점         │
└─────────────────────────────────────────────────────────────────────────────┘
```

| 영역 | 내용 |
|---|---|
| 제목바 | `<화면명>` (좌상단), 우상단 dev/code 아이콘 |
| 메타-좌 | `Chapter`(예: `03. 상품 담기`), `Screen Path`(Case 1/2/3… 진입 경로) |
| 메타-우 | `Application to`(APP·MOBILE WEB 칩), `Screen ID` |
| Description | `Num \| Description` 표 (콜아웃 정의) |
| 캔버스 | 폰 목업 흐름 + 콜아웃 + 화살표 + 인터랙션 점 |

## 2. Screen ID 포맷 (고정 포맷 / 값은 추론·되묻기)

기본 포맷: **`{PRJ}-{PLAT}-{CH}-{TYPE}-{SEQ}`**  (예: `PRD-MO-1-PG-001`)

| 토큰 | 의미 | 예 |
|---|---|---|
| PRJ | 프로젝트 코드 | `PRD` |
| PLAT | 플랫폼 | `MO`(Mobile Web) / `AP`(App) |
| CH | 챕터 번호 | `1`, `3` |
| TYPE | 화면 유형 | `PG`(Page) / `PO`(Popup·Bottomsheet) / `MD`(Modal) |
| SEQ | 3자리 일련번호 | `001` |

- **포맷·토큰 의미는 PROJECT.md `## 화면설계서` 섹션에서 확정** (없으면 위 기본값).
- SEQ는 같은 Chapter 내 마지막 번호 +1로 **자동 증가**. 충돌·애매하면 **1회 되묻기**.

## 3. 색·기호 의미 (고정 — 매번 같음, 되묻지 말 것)

| 색/기호 | 의미 |
|---|---|
| 🟧 오렌지 | 콜아웃 번호 배지(0,1,2…), 영역 지시 브래킷, 흐름 점선 화살표, `※Mobile web 헤더` 점선 박스 |
| 🟩 초록 | `※` 참조주 (다른 설계서·가이드 링크). Description·화면 내 주석 |
| 🔵 블루 | 인터랙션 점(탭 타겟), 현재 포커스 화면 테두리 |
| 🟥 핑크 칩 | `APP` |
| 🟨 탄/옐로 칩 | `MOBILE WEB` |
| ⬜ 회색 | 목업 크롬(status bar·nav·플레이스홀더 이미지). 비활성 화면은 디밍 |

> 목업 화면 자체는 **회색(디자인시스템 neutral)**, 주석·콜아웃만 위 지정색. (목업에 유채색 금지)

## 4. 콜아웃 규칙

- 인터랙션/동작이 있는 영역마다 **오렌지 원형 번호 배지**(0부터) + 해당 영역으로 브래킷.
- 각 번호는 Description 표의 **같은 번호 행**과 1:1.
- 단순 참조주는 번호 대신 **`※`(초록)** 행으로.

## 5. Description 표 작성 규칙

- 컬럼: **`Num` | `Description`**
- `Num`: `※` 또는 `0,1,2…`
- `Description`:
  - 동작을 **트리거 → 결과** 형태로: `헤더 [x] 선택 시 바텀시트 닫힘`
  - 여러 동작은 불릿
  - 다른 문서 의존은 **초록 `※`주**: `※ 자세한 정의는 00. COMMON GUIDE 참고`
  - 추정 금지 — 동작이 불명확하면 사용자에게 확인

## 6. 목업 컨벤션 (모바일 기준)

- 상단 `Status bar`
- 헤더: `※Mobile web 헤더`(오렌지 점선 박스) — back / 타이틀 / 장바구니 아이콘
- 캐러셀 등 `1/N` 인디케이터
- `상품 대표 이미지` 등 회색 플레이스홀더
- 바텀시트/모달: auto-layout (`가로 Fixed, 세로 Fill Container`)
- 하단 nav bar (3버튼)
- 흐름: 화면 간 **오렌지 점선 화살표**, 탭 지점에 **블루 점**
- 흐름 상 비활성/뒷장 화면은 **디밍 + "※…정의는 뒷장 참고"**

## 7. 명시 vs 되묻기 (핵심 — 과잉 질문 금지)

| 항목 | 처리 |
|---|---|
| 레이아웃·색·표 컬럼·목업 크롬·ID 포맷 | **명시(고정)** — 묻지 않음 |
| 이번 화면의 **플로우/스텝·화면명** | **되묻기** (1회): "옵션선택→사이즈→담기 3스텝 맞나요?" |
| Screen ID 자리값 | **추론 우선**, 애매할 때만 1회 되묻기 |
| Description 동작 내용 | 에이전트가 **초안** → 사용자 검토 |

> 색·양식까지 매번 물으면 사용자가 피곤하고 결과가 들쭉날쭉해진다. (v0.5.1 "임의 질문 금지" 철학과 동일)
