# 완료 전 자가 점검 (Self-Check)

작업을 "완료"라고 선언하기 **전에** 아래를 확인한다.
하나라도 통과하지 못하면 완료를 선언하지 말고 먼저 수정한다.

훅(protect-files, detect-hardcoded, enforce-grayscale)이 작업 중 "막는" 것이라면,
이 스킬은 마무리 시점에 산출물을 "스스로 돌아보는" 것이다. 둘은 다른 시점을 지킨다.

## 1차 — 자동 채점 (권장)

Eval 채점기를 돌려 객관적 수치를 확인한다. 에이전트를 재호출하지 않으므로 API 비용이 없다.

```bash
pnpm eval
```

- CRITICAL 0건 + 기준선 대비 하락 없음 → 2차 체크리스트로.
- CRITICAL이 1건이라도 있거나 점수가 기준선보다 떨어졌으면 → **완료 선언 금지.**
  리포트의 위반 항목을 먼저 고친다.
- **"⚠️ 결과 불완전 / 핵심 검사 미검증"이 뜨면 → 완료 선언 금지.** 대비(T1)를 못 돌린 채 나온 "0건"은 통과가 아니다. `/init`으로 토큰을 먼저 생성해 T1이 실제로 채점되게 한 뒤 다시 `pnpm eval`을 돌린다.

훅 자체를 수정했거나 가드 동작이 의심되면 훅 회귀 테스트도 돌린다 (Write/Edit/MultiEdit 입력 shape를 실제로 먹여본다):

```bash
pnpm test:hooks
```

## 2차 — 수동 체크리스트

Eval을 못 돌리는 상황이거나 보강 확인이 필요할 때:

1. [ ] **대비** — 새/수정한 색상 페어가 WCAG AA(일반 4.5:1, UI·큰 텍스트 3.0:1)를 통과하는가?
2. [ ] **하드코딩** — 컴포넌트에 raw hex, `px`, Tailwind 색상 클래스(`bg-blue-500` 등)가 0건인가?
3. [ ] **시맨틱 alias** — 컴포넌트가 raw scale(`--color-brand-500`) 대신 alias(`--color-primary` 등)를 거치는가?
4. [ ] **와이어프레임** — `src/wireframes/`에 시맨틱 색상이 새지 않고 회색만 쓰는가?
5. [ ] **다크모드** — 라이트에서 통과한 페어가 다크에서도 동등성을 지키는가? (`pnpm eval` 또는 `/qa-a11y --check-dark-parity`)
6. [ ] **접근성** — 새 이미지/아이콘에 `alt` 또는 `aria-label`이 있는가?
7. [ ] **Storybook** — 새 컴포넌트에 스토리(Default·Variants·Sizes·States·Matrix)가 생성됐는가?

## 실패 시 행동

- 대비 실패 → semantic alias를 한 단계 어두운 shade로 변경하거나 OKLCH L값 조정. 한 줄 수정으로 전체 반영.
- 하드코딩 발견 → 해당 값을 토큰 변수로 치환.
- 와이어프레임 색상 누설 → 회색 토큰으로 되돌린다. 색을 쓰고 싶다면 디자인 단계로 넘어갈 신호다.

## 원칙

완료 선언은 "내가 다 했다"가 아니라 "검증을 통과했다"는 뜻이다.
이 한 단계가 사람의 QA 부담을 줄이고, 같은 실수가 반복되는 것을 막는다.
