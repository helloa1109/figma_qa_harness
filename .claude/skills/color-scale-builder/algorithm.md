# OKLCH Color Scale Algorithm

> SKILL.md의 알고리즘 절차 상세 설명. 일반 사용 시에는 안 봐도 됨.
> 알고리즘 수정/디버깅이 필요할 때만 참조.

## 왜 OKLCH인가

| 색공간 | 문제 |
|---|---|
| HSL | "같은 L 값인데 노랑은 밝고 파랑은 어둡게 보임" — 지각적 비균일 |
| RGB | 직관적이지 않음 + 비균일 |
| LAB | 균일하지만 HCL(hue-chroma) 분리가 직관적이지 않음 |
| **OKLCH** | **지각적으로 균일** + **밝기·채도·색상 분리** + Tailwind v4/CSS L4 표준 |

## 변환 파이프라인

```
입력 hex (예: #3b82f6)
   ↓ hexToLinearRGB
[0.0556, 0.2227, 0.9387]    (linear sRGB)
   ↓ linRGBtoOklab
[L=0.58, a=-0.044, b=-0.187]  (OKLab)
   ↓ oklabToOklch
[L=0.58, C=0.19, H=259.8°]    (OKLCH)
```

## L 램프 설계 근거

11단계 L 값:
```
[0.97, 0.93, 0.85, 0.76, 0.67, 0.58, 0.50, 0.42, 0.33, 0.25, 0.15]
 50   100   200   300   400   500   600   700   800   900   950
```

- **L=0.97 (50)**: 거의 흰색이지만 살짝 색조감. 카드 배경, 매우 옅은 강조.
- **L=0.58 (500)**: base. 입력 컬러와 가장 유사한 위치.
- **L=0.15 (950)**: 거의 검정에 색조감. 다크모드 헤더, 강한 대비 텍스트.

램프 간격은 **양끝 가파름 + 중간 완만**:
- 50→100: ΔL=0.04 (양끝은 작은 변화도 크게 보임)
- 400→500: ΔL=0.09 (중간은 큰 변화로 단계감 확보)
- 900→950: ΔL=0.10

## C 비율 설계 근거

```
[0.10, 0.25, 0.50, 0.75, 0.90, 1.00, 0.95, 0.85, 0.65, 0.50, 0.30]
 50   100   200   300   400   500   600   700   800   900   950
```

- **500에서 최대 채도** (1.00) — 사용자가 입력한 hex의 chroma 그대로
- **양끝(50, 950)**은 채도 크게 줄임 — 안 그러면 50이 형광 핑크처럼 보이고, 950이 새까만 보라처럼 변함
- **400-600 부근에서 채도 유지** — 그래야 brand 컬러의 정체성이 유지됨

## sRGB Gamut 클램프

OKLCH 공간에서는 표현 가능하지만 sRGB 디스플레이에 못 그리는 영역이 있음 (특히 형광 컬러).
간이 휴리스틱: `C > 0.37`이면 0.37로 클램프. 정확한 클램프는 CSS 브라우저가 알아서 처리.

## Neutral 스케일이 별도인 이유

회색은 chroma=0이라 hue가 의미 없음. 그리고 회색은 **눈으로 가장 민감하게 단계 차이를 봄**.
따라서 별도 L 램프 사용:
```
[1.0, 0.985, 0.96, 0.92, 0.86, 0.7, 0.55, 0.44, 0.37, 0.27, 0.2, 0.13, 0]
  0    50   100  200  300  400  500  600  700  800  900  950  1000
```

`0`과 `1000` 추가는 진짜 흰색/검정 anchor.
